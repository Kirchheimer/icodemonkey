SAM.gov Platform Cleanup

Technical Architect · REI Systems · 2025

Recent platform review work on the SAM.gov deployment uncovered systemic configuration drift that manifested as intermittent container restart spikes, high cross-service latency, and rising host resource allocation. The platform runs in a large, Rancher-managed container environment and required cross-team access to logs and configuration state to diagnose.

Symptom & Impact

  • Symptom: Intermittent container restart spikes and race conditions across multiple micro-services
  • Impact: Operations responded by increasing host resources over six months — masking the real cause and inflating platform cost
  • Contributing noise: Newly migrated micro-services were initially (and incorrectly) blamed for the increased load

The Hidden Issue

Untracked cache-timeout changes (introduced during an upstream Spring Boot upgrade) caused cache entries to persist longer than expected, producing a slow six-month creep of unexpired values. This in turn increased memory pressure and produced downstream timeouts and retries.

Investigation Approach

  • Negotiated cross-vendor log access; aggregated logs and metrics spanning application, caching, and orchestrator layers
  • Correlated request traces, access logs, and Redis / Spring cache metrics to rule out expected traffic surges
  • Wrote targeted scripts to inspect cache contents and TTL distributions to verify un-expired entries and quantify growth

Findings & Actions

  • Identified the Spring Boot upgrade that changed a cache-life variable, increasing default cache TTLs and preventing expected eviction
  • Implemented TTL corrective patches and tuned cache eviction policies; added guardrails to ETL and background jobs to avoid cache amplification
  • Introduced idempotent operational scripts to prune stale caches and added monitoring alerts for cache TTL drift
  • Improved centralized logging/alerting so dev and ops teams could see cache-level failures without siloed access

Results (before → after)

  • Container restart spikes: ~40 spikes → 4–5 expected scheduled restarts per day
  • Cross-service latency: >80% reduction in peak latency spikes
  • Redis daytime latency: dropped to match (or improve upon) off-peak levels
  • Normal daily traffic surges: ceased producing system-wide latency events
  • Host resource allocation: ~97% reduction in emergency resource overprovisioning

Note: This case highlights the value of platform-level audits, cross-team log visibility, and guarding config drift during upstream dependency upgrades.

GSA Automated Testing Program

Technical Architect · REI Systems · 2024 – 2025

Implemented a comprehensive automated testing program for multiple GSA applications to improve release velocity, reduce regressions, and centralize quality metrics across teams.

Scope

End-to-end coverage including unit, integration, E2E, accessibility, performance, and visual regression testing across web applications and APIs.

Approach

  • Adopted a test-pyramid strategy — fast unit/integration tests prioritized; targeted E2E suites for critical user journeys
  • Containerized test runners and integrated automation into CI pipelines — every PR plus nightly for broader coverage
  • Implemented accessibility checks (axe-core, Lighthouse), browser-driven E2E (Playwright, Puppeteer), and API integration tests covering service contracts and error paths
  • Built synthetic performance checks and load smoke tests to catch response-time regressions pre-deploy
  • Automated test data management: seeded fixtures, idempotent environment teardown, feature-flagged test targets

Tooling & Pipelines

  • Test frameworks: Playwright, Puppeteer, Jest / Mocha, pytest (per service)
  • Accessibility & performance: axe-core, Lighthouse, Lighthouse CI
  • CI/CD: GitHub Actions and Jenkins pipelines; parallelized runners to reduce wall time
  • Reporting: Automated HTML reports, coverage artifacts, dashboarded alerts for test failures and regressions

Implementation Highlights

  • Containerized browser runners for consistent E2E execution across environments
  • Parallelized E2E suites — selective smoke tests on PRs, full suites on nightly runs
  • Stability guards: flaky-test isolation, retry policies with backoff, failure-triage automation to classify flaky vs. deterministic failures
  • Integrated test-failure insights with centralized logging and tracing to speed root-cause analysis

Outcomes & Impact

  • Faster PR feedback loops, earlier defect detection, less rework during code review
  • Measurable drop in post-deploy regressions and production incidents
  • Reduced manual regression testing effort; faster, more reliable release cadence
  • Improved accessibility posture — automated checks prevent regressions in key user flows

Next Steps

  • Add quantitative metrics to reports (incident counts, MTTD, test-flakiness rates) on a weekly quality dashboard
  • Expand synthetic load tests to include longer-duration soak tests for capacity planning
  • Continue investment in test-data tooling to improve deterministic outcomes and reduce flakiness

ThinkSheet v2 Modernization

Lead Platform Architect & Full-Stack Engineer · 2024 – 2026

The Conceptual Work & Legacy Context

THINKSheet Systems was founded in 1993 by William Stahl (a quantitative legal and real estate investment banking expert) and Dr. Roy Freedman (a pioneer in financial technology and architect of the NYSE Market Surveillance System). The platform was built to operationalize a groundbreaking paradigm: Optimized Professional Intelligence (PI) + Artificial Intelligence (AI).

While typical financial platforms efficiently model quantitative variables, THINKSheet was designed to capture, model, and visualize the complex qualitative drivers that dictate high-stakes institutional decisions. Its conceptual origins trace back to complex economic models like the 1991 West Point Privatization Project, later expanding into country allocation apps for JPMorgan, research tools for Morgan Stanley, and corporate event-driven investment analysis.

The original system combined advanced quantitative structures (originally built on early AI engines like InduceIT) with sprawling, custom Excel-based modeling ecosystems and Bloomberg Equities app integrations. Over decades of operation, this dense layer of domain intelligence became bottlenecked by its legacy delivery mechanisms: fragile desktop Excel runtimes, siloed local environments, and specialized terminal licensing. The objective of ThinkSheet v2 was to extract this institutional intelligence from its legacy desktop bounds and rebuild it as a secure, scalable, enterprise-grade cloud application without losing the nuance of its multi-variant qualitative scoring.

Modernization - Core Stack

  • Frontend: Vue 3, Vite, Pinia, Tailwind CSS
  • Backend: Node.js, Express API
  • Data Layer: PostgreSQL, Prisma ORM
  • Ops / Deployment: Docker, Docker Compose, Nginx, Cloudflare Tunnel, Linode VPS

Engineering & Architecture Work

  1. Re-Architecting the Proprietary Engine: Translated decades of complex, macro-heavy financial and qualitative evaluation models out of legacy desktop runtimes into an isolated Node.js calculation backend. Designed a highly relational schema in PostgreSQL via Prisma to manage dense, multi-variant project datasets, tracking compounding qualitative weights, user-defined risk profiles, and historical models. Built role-based access control (RBAC) and enterprise user management to allow institutional clients to safely isolate proprietary investment evaluations while enabling collaborative team editing.
  2. Modernizing the Analytical Interface: Engineered a high-performance Vue 3 Single Page Application (SPA) utilizing Vite for optimized bundling and Pinia for reactive, centralized state management. Replicated the dense data density of the legacy terminal apps in a fluid, responsive web UI, allowing analysts to tweak qualitative sliders and instantly visualize shifting risk/reward profiles.
  3. Containerized Infrastructure & Zero-Trust DevOps: Architected a clean, multi-container Docker Compose stack separating frontend, API backend, and database layer to ensure complete parity between local development and production environments. Provisioned and hardened a Linode Linux VPS, leveraging Nginx as a reverse proxy routed through fixed internal service ports. Secured enterprise web access via Cloudflare Tunnels, entirely exposing the platform to authorized traffic without opening vulnerable, public inbound firewall ports on the host system.

Architectural Outcomes

  • Zero-Footprint Access: Migrated a complex enterprise financial tool from a rigid, desktop-bound installation to a modern web application accessible securely from any browser.
  • Eliminated Environment Drift: Containerization dropped environment-specific configuration bugs to zero, providing predictable, atomic deployments.
  • Robust Security Posture: By coupling Cloudflare Tunnels with isolated database containers, the surface area for network attacks was minimized to zero open inbound public ports, meeting strict institutional expectations.

Modernizing Developer Environments for GSA SAM.gov

Technical Architect · REI Systems · GSA IAE / SAM.gov

Executive Summary

The General Services Administration's System for Award Management (SAM.gov) is a critical pillar of federal procurement, processing billions of dollars in transactions. Tasked with supporting this vital infrastructure under the GSA COMET blanket purchase agreement, REI Systems faced severe delivery bottlenecks rooted in legacy, decentralized developer environments. Developers relied on traditional, government-issued Windows machines running localized XAMPP stacks — a configuration plagued by configuration drift, slow hardware distribution, and environmental mismatches.

To overcome these challenges, a breakthrough cloud migration project was executed. The legacy architecture was replaced with a scalable Amazon Web Services (AWS) Cloud WorkSpaces environment integrated with a custom-engineered Docker containerization layer. By providing a virtual developer environment that perfectly mirrors the AWS production ecosystem — including local Apache, legacy-decoupled PHP engines, and dedicated Apache Solr search environments — the project eliminated local vs. production mismatching.

Key Performance Indicators (KPIs) Achieved

  • Developer Onboarding Time: Reduced from 2 weeks to 2 hours.
  • Environment Configuration: Slashed from a week-long manual setup to a single-command automated spin-up.
  • Issue Diagnosis: Achieved near-instantaneous debugging and isolation of complex system bottlenecks.

The Legacy Environment: Challenges & Bottlenecks

Prior to the migration, engineers developed locally on physical, agency-issued Windows machines. This setup introduced compounding operational inefficiencies:

  • Hardware-Bound Onboarding: Bringing a new developer onto the project required shipping physical, government-managed laptops. Delays in equipment transfer, security provisioning, and agency credentialing routinely burned the first two weeks of an engineer's contract.
  • The "XAMPP Fragility" & Configuration Drift: Replicating the SAM.gov enterprise stack using localized XAMPP installations on Windows was notoriously unstable. Configuring the platform took up to a week per developer, requiring granular, error-prone manual tweaks to php.ini, Apache vhosts, and environment variables.
  • Legacy PHP Version Lock-ins: The underlying monolithic components of the legacy application suffered from strict language version dependencies. Managing co-existing legacy PHP installations alongside modern dependencies on a single Windows OS frequently caused local system conflicts.
  • Parity Mismatches (Local vs. Production): Developing on Windows (NTFS file systems, Windows-specific binaries) while targeting a Linux-based AWS enterprise production cloud introduced systemic "works on my machine" bugs. Edge-case bugs, permission errors, and routing idiosyncrasies were frequently caught only after deployment.

The Solution: AWS WorkSpaces + Core Docker Architecture

The core of the modernization project lay in decoupling the developer environment from physical hardware and shifting the entire operating context into a secure, FedRAMP-compliant virtual desktop infrastructure (VDI), standardizing the codebase layer using containerization.

Architectural Breakdown

  1. Amazon WorkSpaces Virtual Desktop: Developers use lightweight cloud desktops accessed securely via standard PIV/CAC smartcard authentication. This eradicated physical device shipping; new instances are now provisioned via AWS identity pools in minutes.
  2. Core Docker Container Project: Rather than hosting application services natively inside the VDI instance OS, a tightly integrated Docker network blueprint was engineered. This containerized network encapsulated the entire runtime stack.
  3. True Linux-to-Linux Parity: Because Docker runs natively on a Linux engine (or via isolated virtualization layers on the WorkSpace host), the developer's execution environment matches the exact kernel characteristics, file system rules, and memory management of the live AWS production target.
  4. Isolated Version Encapsulation: Strict legacy PHP dependencies were isolated inside specific immutable Docker images. Developers can run disparate legacy blocks simultaneously without risking system-wide library pollution.

High-Value Impact: Mirrored Solr Environments & Rapid Debugging

The most significant engineering victory resulting from this architectural shift occurred within the platform's search architecture. SAM.gov relies heavily on Apache Solr for large-scale enterprise data queries, filtering, and indexing.

Accelerated Solr Troubleshooting

Under the old paradigm, running a heavy, multi-core Solr cluster on a local Windows machine was unfeasible due to memory constraints and configuration mismatches. When production Solr bottlenecks occurred, developers were left "blind," forced to look at remote logs or safely guess the root cause.

By packing an exact, mirrored Apache Solr container instance directly into the core Docker setup, engineers gained a fully functioning, standalone Solr sandbox matching production data schemas.

  • Isolate & Conquer: Developers can reliably simulate production-scale workloads, trace slow-running query parameters, and test indexing modifications locally without impacting active stages.
  • Bottleneck Diagnoses: Intermittent timeouts, race conditions, and heavy indexing leaks that previously took weeks of distributed log analysis were isolated, replicated, and patched in hours.

Business and Operational Outcomes

The realization of this cloud-native developer experience yielded immediate dividend returns for the GSA IAE team and REI Systems:

Conclusion

By engineering a container-first paradigm within AWS WorkSpaces, the SAM.gov development lifecycle was successfully untangled from local hardware limitations. The project proved that modernizing the developer experience is just as critical as modernizing application architecture. The result is a highly agile, highly scalable federal engineering workspace built to sustain continuous delivery.
